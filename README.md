# 0xkey Enclave KeyOps Skill

Agent skill / runbook package for operating 0xkey Nitro enclave key ceremonies.
This repository ships **four role skills**, one per ceremony seat. Each role
skill is fully self-contained so it can be installed independently and only
exposes the actions that role is actually allowed to take.

## Skills in this repository

| Skill name | Slash command (OpenClaw, sanitized) | Owns |
|------------|-------------------------------------|------|
| `0xkey-keyops-coordinator` | `/0xkey_keyops_coordinator` | Manifest generation / envelope, deploy render/apply, boot-genesis / boot-standard / post-share, K8s overlay, attestation, verify, key-forward |
| `0xkey-keyops-manifest` | `/0xkey_keyops_manifest` | Review bundle verify, `approve-manifest`, approvals bundle packaging |
| `0xkey-keyops-share` | `/0xkey_keyops_share` | Share-request bundle verify, `ceremony share-extract`, `proxy-re-encrypt-share`, wrapped-shares bundle |
| `0xkey-keyops-builder` | `/0xkey_keyops_builder` | qos_client / qOS release / pivot binaries / pivot hashes / ECR digests / builder-handoff |

Each role skill is `disable-model-invocation: true`, so Cursor / Codex /
Claude Code only load it when the user names it explicitly (or invokes the
matching OpenClaw slash command).

`core/` is the only place to edit shared content. Per-role copies in
`skills/<role>/` are byte-identical to `core/` and are regenerated by
`tools/sync-skills.py` — never hand-edit a per-role copy.

## keyops binary (zero runtime dependencies)

Each GitHub Release publishes a self-contained `keyops` binary built with
PyInstaller — Python 3.12 is bundled inside, so callers need no Python
runtime on their machine.

| Platform | Release asset |
|----------|---------------|
| macOS arm64 | `keyops.darwin-arm64` |
| Linux amd64 | `keyops.linux-amd64` |

### Quick install (verified)

```bash
# Example: macOS arm64
curl -fLO https://github.com/0xkey-io/enclave-keyops-skills/releases/latest/download/keyops.darwin-arm64
curl -fLO https://github.com/0xkey-io/enclave-keyops-skills/releases/latest/download/keyops.darwin-arm64.sha256
shasum -a 256 -c keyops.darwin-arm64.sha256
install -m 0755 keyops.darwin-arm64 ./bin/keyops

# Then use it:
./bin/keyops --version
./bin/keyops doctor coordinator
./bin/keyops init --role coordinator ...
```

### Auto-fetch via skill script

If you already have Python available (or are running from source), the
`fetch_keyops.py` helper handles platform detection, download, and SHA256
verification automatically:

```bash
python3 scripts/fetch_keyops.py --out ./bin/keyops
```

Or through the unified entry point if you already have a `keyops` binary:

```bash
keyops fetch-keyops --out ./bin/keyops
```

## Install / use with agents

Pick the one role skill that matches your seat in the ceremony. Do not install
or load multiple role skills in the same operator session unless you are
maintaining the repository itself.

### Recommended: install from GitHub

Use the public repository path directly:

```bash
npx skills add 0xkey-io/enclave-keyops-skills --skill 0xkey-keyops-coordinator
npx skills add 0xkey-io/enclave-keyops-skills --skill 0xkey-keyops-builder
npx skills add 0xkey-io/enclave-keyops-skills --skill 0xkey-keyops-manifest
npx skills add 0xkey-io/enclave-keyops-skills --skill 0xkey-keyops-share
```

Choose exactly one command for your role:

- Coordinator: `0xkey-keyops-coordinator`
- Builder / Release operator: `0xkey-keyops-builder`
- Manifest Set member: `0xkey-keyops-manifest`
- Share Set member: `0xkey-keyops-share`

If your agent supports platform targets, add the agent selector flags it
documents (for example `-a cursor`, `-a codex`, `-a claude-code`, or
`-a openclaw`). The repository path stays the same:

```bash
npx skills add 0xkey-io/enclave-keyops-skills --skill 0xkey-keyops-manifest \
  -a cursor -a codex -a claude-code -a openclaw
```

### Give this URL to your agent

You can hand an agent this URL and ask it to install the matching role skill:

```text
https://github.com/0xkey-io/enclave-keyops-skills
```

Expected agent behavior:

1. Ask which ceremony role you are operating as: Coordinator, Builder,
   Manifest Set member, or Share Set member.
2. Install the corresponding skill with
   `npx skills add 0xkey-io/enclave-keyops-skills --skill <role-skill>`.
3. Do not guess the role and do not install all four skills unless explicitly
   asked to prepare a maintainer / test environment.
4. After install, start from the matching `SKILL.md` and enforce that role's
   action whitelist.

### Local development install

Use a symlink only when you are developing this repository and want installed
skills to track `git pull` automatically. From inside the cloned repo:

```bash
REPO_ROOT="$PWD"

install_skill() {
  local skill_name="$1"
  local agent_dir="$2"
  mkdir -p "$agent_dir"
  ln -sfn "$REPO_ROOT/skills/$skill_name" "$agent_dir/$skill_name"
}

# Example: a Manifest Set member on Cursor (user-level)
install_skill 0xkey-keyops-manifest "$HOME/.cursor/skills"

# Example: a Coordinator on Codex + Claude Code (user-level)
install_skill 0xkey-keyops-coordinator "$HOME/.codex/skills"
install_skill 0xkey-keyops-coordinator "$HOME/.claude/skills"
```

`ln -s` keeps each installed skill in sync with `git pull` without copying.
Use `cp -R` instead if your agent or security policy rejects symlinked skill
directories.

OpenClaw's `skills.load.allowSymlinkTargets` defaults to off for workspace /
project-agent roots; symlinks under `~/.openclaw/skills/` and
`~/.agents/skills/` are allowed by default but the SKILL.md realpath must
stay inside its resolved skill directory.

### ClawHub install (OpenClaw)

```bash
openclaw skills install <skill-slug>
```

`<skill-slug>` is the ClawHub-published slug for the desired role skill, which
matches the `name:` field (e.g. `0xkey-keyops-manifest`).

### Project-level install

Use project-level install only when a team intentionally pins this skill via
the consuming project's git history. Prefer the agent's normal install command
if it supports project-level targets; otherwise copy the desired
`skills/<role>/` directory into that agent's project skill directory. A
project-pinned version should take precedence over user-level installs.

And prompt:


## Operator usage

Start by telling the agent your exact role. Each line below loads the matching
role skill directly; no role-router round-trip needed.

```text
我是 0xkey KeyOps 的 Builder / Release 角色。
请使用 0xkey-keyops-builder skill，只执行 Builder 角色流程。
```

```text
我是 0xkey KeyOps 的 Deployment Coordinator。
请使用 0xkey-keyops-coordinator skill，只执行 Coordinator 角色流程。
```

```text
我是 0xkey KeyOps 的 Manifest Set 成员，Coordinator 分配给我的 alias 是 <alias>。
请使用 0xkey-keyops-manifest skill，只执行 Manifest Set member 角色流程。
```

```text
我是 0xkey KeyOps 的 Share Set 成员，Coordinator 分配给我的 alias 是 <alias>，
member-index 是 <n>。
请使用 0xkey-keyops-share skill，只执行 Share Set member 角色流程。
```

If a user opens a conversation without naming their role (or with a
cross-role utility action like "verify this bundle"), ask which seat they
have before invoking any role skill. Routing logic belongs in the
conversation, not in a separate skill.

For longer per-role start prompts, see
[`core/references/operator-prompts.md`](core/references/operator-prompts.md)
(also present inside each role skill).

## Safety rules

- Never commit or share `.secret`, `.share`, wrapped-share plaintext, DR
  secret, or extracted sensitive ceremony directories.
- Member long-term key material must live outside the role workdir, preferably
  in an operator-controlled encrypted key vault.
- Manifest / Share members do not need AWS, EKS, or kubectl access; their
  skills do not whitelist any such command.
- Dangerous steps require explicit human confirmation: manifest approval,
  share re-encryption, deploy apply, and post-share. `--yes` does not bypass
  these.

See `core/SECURITY.md` and role documents under `core/references/roles/` (or
the synced copies inside each role skill) before running a ceremony.

## Maintenance

```bash
# Regenerate per-role copies after editing anything under `core/`.
tools/sync-skills.py

# CI check: exit 2 if any role skill has drifted from `core/`.
tools/sync-skills.py --check

# Run the unit test suite (Python 3.11+).
python3 -m unittest discover -t . -s tests -v
```

### Releasing a new version

1. Bump `VERSION` (e.g. `0.4.0`) and update `core/references/release-notes.md`.
2. Run `tools/sync-skills.py` to propagate the version into all `SKILL.md`
   frontmatter.
3. Commit and push, then create a git tag: `git tag v0.4.0 && git push --tags`.
4. The `release.yml` GitHub Actions workflow runs automatically:
   - Builds `keyops.darwin-arm64` (on `macos-latest`) and `keyops.linux-amd64`
     (on `ubuntu-latest`) via PyInstaller.
   - Publishes both binaries plus `.sha256` sidecars and a combined
     `SHA256SUMS` to the GitHub Release.
5. Operators upgrade with `npx skills update 0xkey-keyops-<role>` or by
   re-running `fetch_keyops.py --out ./bin/keyops` to pull the new binary.
